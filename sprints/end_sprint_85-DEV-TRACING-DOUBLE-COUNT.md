## Design (LOCKED — authoritative-window, #72-safe scoped delete)
- **Root (Rule#0, hypothesis CORRECTED):** the dispatch guessed "dedup-key-includes-source" / "source-per-scanner-pass" — BOTH refuted in code (dedup is sha-only; source IS by author-identity on local+remote). The real bug = stale rows orphaned by #84's attribution flip (upsert keys on source → old "other" rows never overwritten).
- **Fix = authoritative-window, ATOMIC (shape (a)):** a scan re-derives [since_day..today] × scanned-repos, so it's authoritative for that window — `store.replace_window(since_day, scanned_repos, aggregates)` DELETEs those rows + upserts the fresh aggregates in ONE transaction. A (date,repo) now all-"you" loses its stale "other". (backend hardened the first cut from delete_window+per-row-upsert → the atomic replace_window per the CRITICAL #72-reinforcement; `delete_window` is KEPT standalone with its own tests.)
- **🔴 #72-INCIDENT-LEVEL SAFETY (the load-bearing part):** (1) SCOPED — the DELETE in replace_window pins BOTH date>=since AND repo IN the explicit scanned set (never blanket, never date-only), params bound. (2) EMPTY scanned set → `if repo_list:` guard skips the DELETE (a 0-commit/unreachable scan can't wipe). (3) ATOMIC — delete + all upserts in ONE txn; ANY upsert raises → `conn.rollback()` → store UNTOUCHED (a flaky/partial scan never leaves a wiped-not-refilled window). Un-scanned repos keep their rows. (the verify-cleanup-scope-delete-not-blanket / #72 portfolio_snapshot lesson, belt-and-suspenders.)
